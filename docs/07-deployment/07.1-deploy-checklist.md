# Checklist de Despliegue — GuideFlow

## Pre-deploy

- [ ] Todos los tests pasan (unitarios + E2E)
- [ ] Backup de la BD de producción (pg_dump)
- [ ] Variables de entorno de producción verificadas
- [ ] Migraciones de Prisma preparadas (`prisma migrate deploy`)
- [ ] SSL configurado en Caddy
- [ ] Stripe en modo live con webhook de producción
- [ ] DNS apuntando al VPS

## Durante el deploy

- [ ] `docker compose pull` → descargar imágenes actualizadas
- [ ] `docker compose up -d --build` → reconstruir y levantar
- [ ] `docker exec app npx prisma migrate deploy` → aplicar migraciones
- [ ] Verificar healthcheck: `curl https://app.guideflow.com/api/health`

## Post-deploy

- [ ] Probar registro de guía en producción
- [ ] Probar pago con tarjeta de test de Stripe
- [ ] Verificar que los emails salen correctamente
- [ ] Revisar logs: `docker compose logs -f app`

## Rollback

Si algo falla:

1. `docker compose down`
2. Restaurar backup de BD: `pg_restore`
3. `docker compose up -d` con la imagen anterior

---

## 🔍 Preguntas de Revisión

- [ ] ¿Hemos hecho backup antes de desplegar?
- [ ] ¿Qué pasa si el despliegue falla? ¿Tenemos rollback?
- [ ] ¿Cómo sabemos que el despliegue ha ido bien?
- [ ] ¿Notificamos a los usuarios de la nueva versión?
- [ ] ¿Hay migraciones de base de datos? ¿Son reversibles?

## 🏢 Preguntas de Operaciones Enterprise

- [ ] ¿Tenemos entornos separados? (dev, staging, producción)
- [ ] ¿Cómo aseguramos que lo que se prueba en staging es idéntico a producción?
- [ ] ¿Hay despliegues automáticos o manuales con aprobación?
- [ ] ¿Cómo hacemos canary deployments o blue-green?
- [ ] ¿Qué métricas de rendimiento monitorizamos? (CPU, RAM, IO, conexiones)
- [ ] ¿Hay alertas proactivas antes de que el sistema caiga?
- [ ] ¿Tenemos un SLA (Acuerdo de Nivel de Servicio) documentado?
