# ✅ Checklist "Modo Paranoico" — GuideFlow

> Antes de dormir tranquilos, respondemos a estas preguntas.

## 🔴 Desastres

- [ ] Si hoy perdemos el servidor, ¿podemos restaurar todo en <4 horas?
- [ ] ¿Hemos probado a restaurar desde esas copias?
- [ ] ¿Tenemos copias de seguridad en al menos 2 ubicaciones diferentes?

## 🔴 Ataques

- [ ] Si un atacante gana acceso a la BD, ¿puede ver contraseñas? (no → están hasheadas)
- [ ] Si un atacante gana acceso al VPS, ¿puede acceder a los secretos? (no → están fuera del repo)
- [ ] Si un guía malicioso modifica URLs, ¿puede ver datos de otros? (no → RLS + validación)

## 🔴 Pagos

- [ ] Si Stripe cae, ¿qué pasa con las reservas en curso?
- [ ] ¿Almacenamos algún dato de tarjeta? (no → Stripe lo gestiona, PCI compliant)
- [ ] ¿Los reembolsos se procesan automáticamente o hay intervención manual?

## 🔴 Equipo

- [ ] ¿Todo el equipo sabe qué hacer en caso de incidente de seguridad?
- [ ] ¿Hay un contacto de emergencia 24/7 si algo explota?
- [ ] ¿Si un miembro clave se va mañana, otro puede mantener el sistema?

## 🔴 Proveedores

- [ ] ¿Nuestros proveedores (Stripe, Hetzner) tienen mejor seguridad que nosotros?
- [ ] ¿Qué pasa si Hetzner cierra o nos echa? ¿Tenemos plan B?
- [ ] ¿Los datos están en la UE? (Hetzner sí, Stripe depende de la config)

## 🔴 La pregunta final

> **¿Qué es lo PEOR que podría pasar? ¿Estamos preparados para eso?**

| Escenario | ¿Estamos preparados? | Plan |
|-----------|---------------------|------|
| Servidor hackeado | ❓ | Backups + rotación de secretos |
| Brecha de datos de clientes | ❓ | Plan de respuesta + notificación GDPR (72h) |
| Stripe nos bloquea la cuenta | ❓ | Migrar a otro procesador (Redsys, Adyen) |
| Guía demanda por error en cobro | ❓ | Terms of Service + seguro de responsabilidad |
| Todo el equipo enferma a la vez | ❓ | Docs suficientes para onboardear a alguien nuevo |

---

## 🚀 Cómo usar esta checklist

1. Cada responsable de área coge las preguntas de su sección
2. Responde **por escrito** en el propio documento `.md`
3. Si la respuesta revela una carencia → crear tarea para solucionarlo
4. Cuando todo esté respondido y resuelto → la sección está "lista para auditoría"
5. Revisión cruzada: otro miembro verifica las respuestas

### Ejemplo de respuesta que buscamos

> **Pregunta:** ¿Cómo evitamos que un guía vea datos de otro modificando IDs?
>
> **Respuesta:** Implementamos RLS en PostgreSQL con política `tenant_id = current_setting('app.current_tenant')`. Además, todas las queries pasan por una capa de servicio que inyecta automáticamente el `tenant_id`. Los IDs en URLs son UUIDs no secuenciales y se validan contra el tenant antes de devolver datos.
