## 📊 Comparación para GuideFlow

| Aspecto | Monolito (actual) | Microservicios |
|---------|-------------------|----------------|
| Complejidad | Baja ✅ | Alta |
| Deployment | Simple ✅ | Complejo |
| Debugging | Fácil ✅ | Distributed tracing |
| Escalado independiente | ❌ | ✅ |
| Equipo necesario | 1-3 devs ✅ | 5+ devs |
| Latencia inter-servicio | 0ms ✅ | 1-5ms por hop |

## 🎯 Decisión: MONOLITO MODULAR

Para los próximos 2 años, mantener monolito Next.js con **modularidad interna**:

```
src/app/
├── api/
│   ├── routes/    # Módulo rutas
│   ├── guides/    # Módulo guías
│   ├── bookings/  # Módulo reservas
│   ├── premium/   # Módulo premium
│   └── stripe/    # Módulo pagos
├── (public)/      # Páginas públicas
└── (dashboard)/   # Páginas privadas
```

## 📊 Cuándo considerar micro-servicios

| Señal | Umbral |
|-------|--------|
| Equipo >10 desarrolladores | Dividir ownership |
| Un módulo necesita escalar 10x | Extraer como servicio |
| Un módulo tiene stack diferente (Python/ML) | Naturaleza diferente |
| Deploy de un módulo rompe otro | Desacoplamiento necesario |

## 🎯 Candidatos a extraer (futuro)

1. **Servicio de mapas/tiles** (heavy I/O, podría ser Go/Rust)
2. **Servicio de IA/ML** (Python, GPU)
3. **Procesamiento GPS** (batch, colas)
