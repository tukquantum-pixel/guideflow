## 🎯 Cuándo shardear

- >100k usuarios y >100GB de datos
- Cuando una sola BD (incluso con read replicas) no puede manejar la carga
- Probablemente NO necesario en los primeros 2-3 años de GuideFlow

## 📊 Estrategia de sharding para GuideFlow

| Opción | Criterio | Pro | Contra |
|--------|----------|-----|--------|
| **Por región geográfica** | Pirineos, Sierra Nevada, etc. | Queries localizadas | Cross-region queries complejas |
| **Por guía/tenant** | guideId | Aislamiento natural | Guías con diferente volumen |
| **Por fecha** | Partitioning temporal | Simple | Solo útil para datos temporales |

## 🎯 Recomendación

1. **Año 1-2**: NO shardear. Optimizar queries + indices + réplicas.
2. **Año 3+**: Evaluar partitioning geográfico si >50GB de datos geoespaciales.
3. **Nunca**: shardear datos de usuario (pocos y críticos).

## 📊 Alternativa: Citus (PostgreSQL distribuido)

```sql
-- Convertir Track en tabla distribuida
SELECT create_distributed_table('tracks', 'region_id');
-- Queries se distribuyen automáticamente por región
```
