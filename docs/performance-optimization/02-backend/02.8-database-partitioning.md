## 🎯 Cuándo particionar

| Tabla | Filas esperadas (año 1) | Particionar | Criterio |
|-------|-------------------------|-------------|----------|
| Activity | ~1.000 | ❌ | Demasiado pequeña |
| Booking | ~10.000 | ❌ | Aún gestionable |
| TrafficData (premium) | ~12M | ✅ | Por mes (timestamp) |
| ActivityFeed | ~500k | ⚠️ | Evaluar en 6 meses |

## ⚙️ Partitioning por rango temporal

```sql
-- Para TrafficData (datos GPS anónimos)
CREATE TABLE traffic_data (
  id UUID DEFAULT gen_random_uuid(),
  h3_index TEXT, lat FLOAT, lng FLOAT,
  timestamp TIMESTAMPTZ DEFAULT now()
) PARTITION BY RANGE (timestamp);

CREATE TABLE traffic_data_2026_03 PARTITION OF traffic_data
  FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

CREATE TABLE traffic_data_2026_04 PARTITION OF traffic_data
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
```

## 📊 Beneficios

- Queries filtradas por fecha solo escanean la partición relevante
- Limpieza de datos antiguos: `DROP TABLE traffic_data_2025_01` (instantáneo)
- Vacuum/Analyze más rápido por partición
