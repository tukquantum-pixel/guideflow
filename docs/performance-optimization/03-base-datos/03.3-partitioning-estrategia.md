# 03.3 – Partitioning Estrategia

| Metadata | Valor |
|----------|-------|
| Versión | 1.0 |
| Última actualización | 2026-03-08 |
| Responsable | DBA |
| Estado | Borrador |

## 📊 Candidatas a particionar

| Tabla | Filas/año (est.) | Tipo | Criterio |
|-------|-------------------|------|----------|
| TrafficData | 12M+ | Range | timestamp (mensual) |
| ActivityFeed | 500k+ | Range | createdAt (trimestral) |
| Logs (futuro) | Variable | Range | timestamp (semanal) |

## ⚙️ Implementación

```sql
-- Tabla padre particionada
CREATE TABLE traffic_data (
  id UUID DEFAULT gen_random_uuid(),
  h3_index TEXT NOT NULL,
  lat FLOAT NOT NULL, lng FLOAT NOT NULL,
  anon_id TEXT,
  timestamp TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (timestamp);

-- Crear particiones automáticamente con pg_partman
CREATE EXTENSION pg_partman;
SELECT partman.create_parent(
  'public.traffic_data', 'timestamp', 'native', 'monthly'
);
```

## 🎯 Limpieza de particiones antiguas

```sql
-- Borrar datos de hace >90 días (instantáneo vs DELETE)
DROP TABLE IF EXISTS traffic_data_2025_12;
```

## 📊 Beneficios

- Query `WHERE timestamp > '2026-03-01'` solo escanea 1 partición
- VACUUM por partición (más rápido)
- DROP TABLE vs DELETE FROM (O(1) vs O(n))
