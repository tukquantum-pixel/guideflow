# 09.5 – Optimización: WebSockets / Live Tracking

## 🎯 Problema

Live tracking envía posición GPS cada 30s. Con 1000 usuarios simultáneos: 1000 conexiones WS + 2000 mensajes/minuto.

## ⚙️ Optimizaciones

| Técnica | Beneficio |
|---------|-----------|
| **Batch de puntos** | Enviar 5 puntos cada 2.5min vs 1 cada 30s |
| **Compresión de coords** | `[42.571234, -1.582345]` → encoded polyline |
| **Heartbeat relajado** | 60s vs 30s para connexiones estables |
| **Fallback SSE** | Si WS no disponible, Server-Sent Events |

## 📊 Estimación de recursos

| Usuarios tracking | Conexiones WS | Mensajes/min | RAM estimada |
|-------------------|---------------|--------------|-------------|
| 100 | 100 | 200 | ~50MB |
| 1.000 | 1.000 | 2.000 | ~200MB |
| 10.000 | 10.000 | 20.000 | Necesita servicio dedicado |

## 🎯 Arquitectura recomendada

```
[App móvil] → [API Gateway] → [Redis Pub/Sub] → [WS Server]
                                    ↓
                              [PostgreSQL] (solo persistencia)
```

- Redis Pub/Sub para distribuir mensajes entre instancias
- WS Server separado del app server (a partir de 1k conexiones)
- Persistir en BD solo al finalizar sesión (no en real-time)
